# mdpad++ — Архитектура

> Подробное описание архитектуры проекта. Читай при работе над задачами,
> затрагивающими архитектуру, движок редактора или хранение состояния.
> Краткая версия — в `CLAUDE.md`.

## Почему Tauri 2, а не Electron

- Бинарник в 10–20 раз меньше (~10 МБ против ~150 МБ).
- Меньшее потребление RAM — критично для "лёгкого" редактора.
- Системный WebView (WebView2 на Windows, WKWebView на macOS, WebKitGTK на Linux).
- Минус: разные WebView на разных ОС → больше тестирования. Принимаем.

## Почему CodeMirror 6, а не Monaco / Lexical / своё решение

- CM6 единственный, у кого декорации, replace-widgets, gutters и виртуализация
  собраны в нужной комбинации для гибрида "raw ↔ rendered по блокам".
- Размер бандла ~400 КБ против ~5 МБ у Monaco.
- Monaco архитектурно заточен под код, а не под mixed rich-text.
- Lexical (Meta) — ближе к rich-text, но нет встроенных line numbers, gutters,
  и переключение raw↔render пришлось бы писать с нуля.
- Своё решение — 6–12 человеко-месяцев на rope, IME, виртуализацию. Не оправдано.

## Inline render: ключевая идея

Документ хранится как обычный Markdown-текст в `EditorState` CodeMirror.
Markdown парсится на блоки (заголовки, параграфы, списки, code-fences и т.д.).
Каждый блок **по умолчанию** отображается через `Decoration.replace` с
кастомным `WidgetType`, который рендерит HTML-представление блока.

По нажатию **F2** на текущем блоке (или двойному клику по виджету) виджет
временно снимается — открывается raw-Markdown с полной поддержкой
редактирования. После потери фокуса / Esc — снова рендер.

Это позволяет:

- сохранить производительность виртуализации (только видимые блоки рендерятся),
- избежать сложностей rich-text engine (исходник всегда плоский MD),
- получить эффект "Typora" без переписывания модели документа.

### Техническая реализация

- `ViewPlugin` строит `DecorationSet` из block ranges.
- Блоки парсятся через `markdown-it` в функции `parseBlocks(source): Block[]`.
- Block-типы: heading, paragraph, list, code_fence, blockquote, table, hr, html_block.
- `RangeSetBuilder` строит декорации.
- Виджет рендерит HTML через `markdown-it.render(raw)`.
- Кэш по хэшу `raw → html` чтобы избегать повторного рендера.
- При больших документах декорации создаются лениво (viewport ± padding).
- `StateField<number | null>` хранит позицию активного (raw) блока.

### Режимы отображения

Три режима (per-document):

- **rendered** (default) — все блоки отрендерены, F2 открывает один блок.
- **mixed** — блок под кареткой автоматически raw, остальные rendered.
- **raw** — весь документ без декораций, чистый Markdown.

Toggle через `Ctrl+E` и кнопку в статус-баре.

## Хранение состояния

- **Открытые документы**, **вкладки**, **настройки**, **последние файлы** —
  в Svelte stores ($state), персистятся в JSON в каталоге Tauri `app_data_dir`.
- **Содержимое файлов** не дублируется в state, источник правды — диск.
  В памяти держим только dirty buffers (несохранённые изменения).
- **Автосохранение** — debounce 2 секунды после последнего ввода, либо при
  потере фокуса окна. Несохранённые изменения дополнительно сбрасываются
  во временный snapshot (`app_data_dir/recovery/`) для recovery после сбоя.

## Diff-полоса (gutter изменённых строк)

- При открытии файла снимаем "baseline" — содержимое с диска.
- На каждый редактирующий transaction CodeMirror диффим текущий буфер с baseline
  построчно (алгоритм Майерса).
- Декорируем gutter: жёлтый — изменено, зелёный — добавлено,
  красный треугольник — удалено.
- При сохранении baseline обновляется, индикация сбрасывается.

## Структура Tauri-команд (Rust)

Все команды в отдельных модулях по доменам:

- `fs_commands.rs` — `read_file`, `write_file`, `pick_open_file`, `pick_save_file`, `pick_folder`, `list_dir`
- `search.rs` — индексация и поиск по содержимому документов
- `watcher.rs` — FSEvent watcher для отслеживания изменений на диске

Все возвращают `Result<T, String>`. Регистрируются в `lib.rs` через
`tauri::generate_handler![]`.

## Производительность

Целевые показатели:

- Открытие 5MB документа < 1 секунда
- Скролл на документе любого размера @ 60fps
- Input latency < 16ms
- F2 toggle < 150ms

Ключевые механизмы:

- Виртуализация CodeMirror 6 — рендерятся только видимые строки
- Ленивый парсинг блоков — только viewport ± padding
- Кэш rendered HTML по хэшу raw-контента блока
- Diff-пересчёт с debounce 100ms
- Поиск в worker / chunked, не блокирует UI
