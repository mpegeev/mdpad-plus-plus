# Linear ↔ GitHub integration + contribution docs

**Статус:** Завершена
**Linear:** MDP-5
**Уровень риска:** standard
**Создана:** 2026-04-16
**Завершена:** 2026-04-16

## Цель

Подключить Linear к GitHub и задокументировать workflow для людей и для Claude Code.

## Критерии приёмки

1. В Linear (Settings → Integrations → GitHub) подключён репозиторий `mdpad-plus-plus`. Свидетельство — подтверждение супервайзера в Linear UI.
2. PR этой задачи с текстом `Closes MDP-5` в описании автоматически связывается с issue MDP-5 в Linear. Свидетельство — связанный PR виден в Linear UI на странице MDP-5.
3. `.github/PULL_REQUEST_TEMPLATE.md` содержит чеклист с полем для Linear-ссылки (`Closes MDP-XXX`).
4. `CONTRIBUTING.md` явно описывает шаги: (a) выбор issue в Linear → (b) ветка `feat/MDP-XXX-slug` → (c) Conventional Commits → (d) PR с `Closes MDP-XXX` → (e) прохождение CI → (f) ревью → (g) merge.
5. Раздел "Работа с Claude Code" в `CONTRIBUTING.md` содержит: ссылки на `CLAUDE.md`, `docs/senar.md`, перечень slash-команд (`/new-task`, `/done`, `/rca`), и правило "один PR = один issue".

## Негативные сценарии

- PR без упоминания MDP-номера — не ломается, просто не линкуется (никаких ошибок CI/integration).
- Упоминание несуществующего MDP-9999 — Linear молча игнорирует, PR не блокируется.
- Ссылка `Closes MDP-XXX` в коммит-сообщении (а не только в PR description) — тоже должна линковаться; проверяется и документируется в `CONTRIBUTING.md`.

**Менять:**

- `tasks/MDP-5.md` (метаданные задачи SENAR — создаётся `/new-task`, обновляется `/done`)

**НЕ трогать:**

- `src/`
- `src-tauri/`
- `CLAUDE.md`
- `DESIGN.md`
- конфиги сборки (`package.json`, `Cargo.toml`, `tauri.conf.json`, lock-файлы)
- `README.md` (область как в Linear; ссылку на CONTRIBUTING — отдельной задачей при необходимости)
- `.github/workflows/` (CI — не предмет задачи)

## Дополнительные ограничения

- Не добавлять зависимостей.
- Не создавать GitHub Issue templates (issues живут в Linear).
- Конфигурация интеграции выполняется супервайзером в Linear UI; задача в коде ограничена docs + PR-шаблоном.
- PR этой задачи сам служит "тестовым PR" для критерия 2 (один PR закрывает задачу и одновременно валидирует интеграцию).

## Решения

**1. PR этой задачи = "тестовый PR" для критерия 2.**
Совмещение работы по задаче и валидации интеграции в одном PR вместо
отдельного throwaway-PR. Нулевой PR-мусор, один прогон CI.

**2. PR-template не трогался — критерий 3 был уже выполнен на main.**
Файл `.github/PULL_REQUEST_TEMPLATE.md` существовал до задачи (создан
ранее — по git blame, в одной из MDP-1/MDP-2). Задача на этапе
`/new-task` ошибочно пометила его как "новый". Исправлено в области
изменений; критерий закрыт верификацией содержимого, а не созданием.

**3. Сокращение раздела "Работа с Claude Code" до one-liner с делегацией.**
Первая версия дублировала 7 правил из CLAUDE.md/SENAR — senar-reviewer
справедливо указал (FAIL[10]) на риск рассинхронизации. Решение —
объявить CLAUDE.md и docs/senar.md единственными источниками истины,
а в CONTRIBUTING.md оставить только то, что требуется критерием 5
(оба линка, "1 PR=1 issue", три slash-команды). Раздел сокращён с
25 до 13 строк.

**4. `tasks/MDP-5.md` явно добавлен в "Менять".**
Шаблон `/new-task` не упоминает сам файл задачи в области, хотя `/done`
его всегда модифицирует. Зафиксировано в области MDP-5; стоит вынести
в улучшение шаблона `/new-task` отдельной задачей.

## Известные проблемы

**1. Расширение области во время работы (правила 2 и 6 SENAR — серая зона).**
Изначально область включала только `.github/PULL_REQUEST_TEMPLATE.md` и
`CONTRIBUTING.md`. Файл `tasks/MDP-5.md` был добавлен в область уже на
шлюзе "Готово" по требованию senar-reviewer (FAIL[1]) — это формально
расширение области без отдельного согласования через супервайзера на
старте. В данном случае согласовано явным решением супервайзера во
время `/done`. Принято как методологическая особенность шаблона
`/new-task`.

**2. Шаблон `/new-task` не учитывает существующие файлы.**
Он маркирует все целевые пути как "(новый)" по умолчанию, что привело
к ошибке формулировки критерия 3. Стоит обновить шаблон — отдельной
задачей по улучшению инструментария SENAR.
