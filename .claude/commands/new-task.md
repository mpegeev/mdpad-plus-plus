# Новая задача SENAR

Имя задачи: **$ARGUMENTS**

Я хочу создать новую задачу. Проведи меня через шлюз "Старт" SENAR.

Задай мне следующие вопросы **по одному**, дожидайся ответа на каждый,
прежде чем переходить к следующему. Не пиши код, не редактируй файлы
кроме `tasks/$ARGUMENTS.md`.

## Вопросы

1. **Цель** — какова цель задачи одним предложением? Если ответ длиннее
   одного предложения — попроси сократить.

2. **Критерии приёмки** — перечисли проверяемые условия. Каждый должен
   быть проверяем независимо. Если ответ слишком абстрактный
   ("работает корректно", "хорошо протестирован") — попроси конкретику
   с ожидаемым поведением и значениями.

3. **Негативный сценарий** — какой минимум один негативный сценарий
   (ошибка, некорректный ввод, граничное условие)? Если пользователь
   говорит "нет таких" — предложи сам 2-3 варианта на основе цели и
   попроси выбрать или дополнить.

4. **Область изменений** — какие файлы/модули можно менять? Какие
   НЕЛЬЗЯ трогать? Если ответ слишком общий ("frontend") — попроси
   конкретные пути.

5. **Уровень риска** — задача затрагивает что-либо из:
   - файловые операции (чтение/запись на диск)
   - пользовательские данные (настройки, recovery)
   - внешние API
   - Tauri ACL / capabilities
   - инфраструктуру/CI
   - производительность на больших документах

   На основе ответа определи уровень: standard / high / critical.
   Если есть сомнение — бери уровень выше. Объясни выбор одной фразой.

6. **Тип задачи (детерминированность)** — относится ли задача к одной из
   категорий, где правильный ответ известен заранее:
   - парсер, валидатор, трансформер, форматтер
   - pure function (без побочных эффектов)
   - store/state machine с явными инвариантами
   - FS-операция с тестируемой логикой (моки на boundary)

   Если ДА — пометь как `детерминированная`. Это влияет на следующий шаг:
   тесты будет писать отдельный субагент `test-writer` независимо от
   разработчика (правило 4 SENAR, защита от MDP-11-style ложных позитивов).

   Если НЕТ (UI визуальный компонент, refactor без изменения поведения,
   docs, инфра-настройки) — пометь как `недетерминированная`, обычный flow.

## После ответов

Создай файл `tasks/$ARGUMENTS.md` со следующей структурой:

```markdown
# $ARGUMENTS

**Статус:** В работе
**Linear:** MDP-XXX (если есть)
**Уровень риска:** <standard|high|critical>
**Тип задачи:** <детерминированная|недетерминированная>
**Создана:** <дата>

## Цель

<одно предложение>

## Критерии приёмки

1. <критерий>
2. <критерий>
   ...

## Негативные сценарии

- <сценарий>

## Область изменений

**Менять:**

- <путь>

**НЕ трогать:**

- <путь или категория>

## Дополнительные ограничения

<если есть: следовать DESIGN.md, не добавлять зависимости, и т.п.>

---

## Свидетельства верификации

<заполняется при закрытии задачи через /done>

## Тупики

<заполняется при закрытии задачи>

## Решения

<заполняется при закрытии задачи>

## Известные проблемы

<заполняется при закрытии задачи>
```

## После создания файла

Покажи супервайзеру содержимое созданного файла.

**Если задача детерминированная** (ответ на вопрос 6) — предложи запустить
`test-writer` субагента **до старта разработки**:

> Задача `$ARGUMENTS` — детерминированная (парсер/валидатор/pure function).
> По правилу 4 SENAR тесты пишет отдельный субагент `test-writer`. Запустить
> его сейчас? Он напишет тесты ТОЛЬКО на основе AC и сигнатур, без доступа
> к будущей реализации. Тесты появятся в проекте и будут падать,
> при этом разработчик не сможет их редактировать.

Если супервайзер согласен — вызови:

> Use the test-writer subagent to write tests for `$ARGUMENTS` based on
> the acceptance criteria in `tasks/$ARGUMENTS.md` and the function/component
> signatures. Place tests in `<path-to-impl-dir>/<name>.independent.test.ts`.
> Do NOT read the implementation file.

**Подтверди что тесты падают — это смысл запуска до разработки.**
Пока они не упали, у нас нет доказательства, что они вообще что-то проверяют
(нарушение правила 3).

1. Прогони новые `*.independent.test.ts` через `npm run test`
   (или `cargo test`).
2. Покажи вывод супервайзеру. Все тесты должны быть **red**.
3. Если хоть один тест **прошёл** до начала разработки — то это говорит о том, что он не делает реальную проверку (или функциональность уже есть). Разберись с супервайзером, не игнорируй.

Запиши в `tasks/$ARGUMENTS.md` короткое свидетельство red-baseline
(сколько тестов, на чём падают — assertions или import). После —
**остановись**.

Если супервайзер НЕ согласен запускать test-writer сейчас (например,
хочет сделать TDD-цикл сам или сразу к разработке) — скажи:

> Задача создана в `tasks/$ARGUMENTS.md`. test-writer пропущен — будет
> запущен на финальном коммите через `/done` шаг 4.5. Учти: это ослабленный
> fallback — red-baseline получить уже не выйдет (реализация будет готова),
> подтвердится только независимая трактовка критериев, а не то, что тест
> поймал бы баг. Когда будешь готов начать работу — дай команду.

И **остановись** — в обоих случаях. Не начинай работу над задачей.
Дождись отдельной инструкции.

**Если задача недетерминированная** — просто скажи:

> Задача создана в `tasks/$ARGUMENTS.md`. Когда будешь готов начать
> работу — дай команду.
